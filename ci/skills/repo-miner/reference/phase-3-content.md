## 分析步骤

### 3.1 动机与定位

阅读以下文件（按优先级）：

1. `README.md` / `README` — 项目自我介绍
2. `docs/` 目录下的设计文档 / 架构文档
3. `CHANGELOG.md` / `HISTORY.md` — 变更历史
4. `CONTRIBUTING.md` — 贡献指南（反映项目治理）
5. `.github/` — Issue/PR 模板（反映工程化程度）
6. 任何 `why-*`, `design-*`, `architecture-*`, `adr-*` 文档

回答核心问题：
- 这个项目要解决什么问题？
- 为什么现有方案不够用？（项目的存在理由）
- 目标用户是谁？

### 3.2 作者视角价值分析

> **前提**：基于上方 Phase 1 上下文中的作者画像和官方文档洞察。
> **跳过条件**：如果 Phase 1 标注"无官方文档/博客，已跳过"且无法从代码/README 推断作者意图，简短说明并跳过。

分析维度：

**① 问题发现视角**：作者是怎么发现这个问题的？
- 来自自身痛点（dogfooding）还是观察他人？
- 是学术研究延伸还是工程实践沉淀？
- 时机选择：为什么是现在，而不是 2 年前或 2 年后？

**② 解法哲学**：作者选择了什么样的解法，背后的价值观是什么？
- 简单 vs 功能完整（Unix 哲学 vs 大而全）
- 性能 vs 易用性的权衡偏向
- 开放 vs 封闭的生态策略
- 对比竞品，作者明确选择了不做什么？

**③ 背景知识迁移**：作者从哪些领域带来了别人没有的 insight？
- 把 A 领域的成熟方案用到 B 领域（跨域移植）
- 用学术研究解决工程问题（理论落地）
- 用之前公司/项目的经验解决新问题

**④ 战略图景**：这个项目在作者更大的规划中处于什么位置？
- 是核心产品还是基础设施？
- 是否有商业化意图（SaaS/托管版/企业版）？
- 开源策略：genuinely open 还是 open-core？

### 3.3 架构与设计决策

**不是罗列模块，而是提炼设计背后的思考。**

步骤：
1. 分析项目目录结构（`find . -maxdepth 3 -type d` 或等效）
2. 阅读入口文件（`main.rs`, `lib.rs`, `index.ts`, `app.py` 等）
3. 识别核心抽象（trait, interface, abstract class, protocol）
4. 阅读配置文件（`Cargo.toml`, `package.json`, `pyproject.toml` 等）
5. 分析依赖选择（为什么选 X 而不选 Y）

**对每个发现的设计决策，用这个格式记录：**

```
决策: [描述]
问题: [它要解决什么问题]
方案: [具体怎么做的]
Trade-off: [这个选择牺牲了什么，换来了什么]
可迁移性: [能否用到其他项目中]
```

**重点关注高价值设计模式（深层 > 表层）：**

| 表层描述（低价值） | 深层分析（高价值） |
|---|---|
| "用了 X 做序列化" | "用 X 的某个特定 feature 解决了 Y 问题" |
| "有文件锁" | "temp+rename 原子写入 + 排他锁，解决并发崩溃恢复" |
| "支持多后端" | "Trait 抽象 + Router 策略链，运行时动态选择 vs 编译期绑定" |

### 3.4 创新点识别

在代码中寻找以下信号：

- **独特的算法或数据结构** — 不是标准库 / 常见 crate 提供的
- **巧妙的 API 设计** — Builder pattern, Type-state pattern, Newtype 等
- **不寻常的架构选择** — 文件即数据库、Git notes 做审计、编译期 feature gating 等
- **跨域知识融合** — 把 A 领域的方法用到 B 领域
- **性能优化技巧** — zero-copy, SIMD, 内存布局优化等

对每个创新点评分：
- **新颖度**（1-5）：这个做法有多少人知道？
- **实用性**（1-5）：能直接用到生产项目吗？
- **可迁移性**（1-5）：是否依赖特定上下文，还是通用适用？

### 3.5 竞品交叉分析

> **跳过条件**：如果 Phase 1 标注"无明显竞品，已跳过"，此节也跳过。

基于 Phase 1 上下文中的竞品清单，对每个主要竞品进行交叉分析。

维度选择原则：选 3-5 个对目标用户**决策最关键**的维度。

```
vs [竞品名]:
  我们更好: [具体在哪里，为什么，有数据支撑吗？]
  竞品更好: [诚实说明，不要回避]
  不同目标: [如果是错位竞争，说明各自适合什么场景]
  用户迁移成本: [从竞品迁移到目标项目需要付出什么代价？]
```

综合竞争结论：
- **差异化护城河**：技术护城河 / 生态护城河 / 信任护城河
- **竞争风险**：最可能被哪个竞品替代
- **生态定位**：在整个技术生态中扮演什么角色

### 3.6 代码质量评估

```bash
cd $LOCAL_PATH

# 测试覆盖
find . -name '*test*' -o -name '*spec*' | grep -v node_modules | grep -v .git | head -20

# 文档质量
find . -name '*.md' -not -path '*/node_modules/*' -not -path '*/.git/*' -exec wc -l {} + | tail -1

# 错误处理模式（根据语言选择）
# Rust: unwrap/expect 滥用检测
grep -rc '\.unwrap()' src/ 2>/dev/null | awk -F: '{s+=$2} END {print "unwrap:", s}'
grep -rc '\.expect(' src/ 2>/dev/null | awk -F: '{s+=$2} END {print "expect:", s}'

# CI/CD 配置
ls .github/workflows/ .gitlab-ci.yml .circleci/ Makefile justfile 2>/dev/null
```

质量信号检查清单：
- [ ] 有测试（单元/集成/E2E）
- [ ] 有 CI/CD 配置
- [ ] 有文档（不仅是 README）
- [ ] 错误处理规范
- [ ] 有 linter / formatter 配置
- [ ] 有 CHANGELOG
- [ ] 有 LICENSE
- [ ] 有示例代码 / examples 目录
- [ ] 依赖版本锁定（lock file）
